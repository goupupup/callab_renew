# 웹 페이지 개발 필수 보안 요구사항 체크리스트 (Baseline)

> 목적: 웹 애플리케이션(웹페이지 포함) 개발/운영 시 **필수적으로** 적용해야 하는 보안 통제 항목을 한 문서로 정리  
> 범위: 프론트엔드, 백엔드(API), 인증/인가, 데이터, 인프라/배포, 운영/모니터링  
> 기준: OWASP ASVS/Top 10, 일반적인 보안 베스트프랙티스 기반 (조항번호 등은 특정 표준에 종속되지 않게 작성)

---

## 0. 기본 원칙 (반드시 준수)
- **최소권한(Least Privilege)**: 사용자/서비스/DB/CI 권한은 필요한 최소로.
- **기본 거부(Default Deny)**: 허용 리스트 기반으로 접근/정책 구성.
- **입력은 불신(Zero Trust Input)**: 모든 외부 입력(헤더/쿠키/바디/쿼리/파일)을 신뢰하지 않음.
- **보안은 설계에 포함(Security by Design)**: 릴리즈 전 보안 점검을 “추가 작업”이 아니라 “필수 게이트”로 둠.
- **민감정보 최소화(Data Minimization)**: 수집/저장/노출을 최소로.

---

## 1. 인증(Authentication) 필수 요구사항
- **강력한 비밀번호 정책**
  - 충분한 길이(예: 12자 이상 권장), 사전/유출 패스워드 차단(가능하면).
  - 로그인 실패 시 **계정 잠금/지연(Throttling)** 적용.
- **다중요소인증(MFA)**: 관리자/운영자/결제/민감데이터 접근 계정은 필수.
- **안전한 세션 관리**
  - 세션/토큰은 **예측 불가한 난수**.
  - 세션 만료(Idle/Absolute) 적용.
  - 로그아웃 시 서버 세션/리프레시 토큰 무효화.
- **계정 복구(비밀번호 재설정)**
  - 재설정 토큰은 단시간 만료, 1회 사용, 강한 랜덤.
  - 재설정 링크/토큰을 로그에 남기지 않음.

---

## 2. 인가(Authorization) 필수 요구사항
- **서버 측 인가 검증은 필수**
  - UI에서 숨겼다고 안전하지 않음. 모든 API에 대해 서버에서 권한 체크.
- **RBAC/ABAC 명확화**
  - 역할(Role)과 권한(Permission) 매트릭스 정의.
  - 관리자 기능은 별도 엔드포인트/가드로 분리.
- **IDOR 방지**
  - 리소스 ID를 통한 접근 시 “해당 사용자의 소유/권한”을 항상 검증.
- **멀티테넌트(고객사 분리)**
  - tenant_id 스코프 강제(쿼리, 캐시, 로그, 스토리지 모두).

---

## 3. 입력 검증 & 인젝션 방어 (OWASP Top 수준)
- **서버 측 입력 검증**
  - 타입/범위/길이/패턴 검증(Allowlist 권장).
  - JSON Schema / DTO validation 등 구조적 검증 적용.
- **SQL/NoSQL/Command Injection 방지**
  - SQL: Prepared Statement/ORM 사용, 문자열 조립 금지.
  - OS Command: 쉘 호출 최소화, 불가피 시 인자 분리/이스케이프.
- **XSS 방지**
  - 출력 인코딩(HTML/Attribute/JS/URL context 별).
  - 위험한 `innerHTML`/template raw rendering 지양.
  - 사용자 입력을 그대로 렌더링하지 않기.
- **SSRF 방지**
  - 서버에서 외부 URL fetch 필요 시:
    - 허용 도메인/대역 allowlist
    - 내부 IP/메타데이터 엔드포인트 차단
- **Path Traversal 방지**
  - 파일 경로는 서버에서 매핑/정규화 후 허용 경로만.
- **Deserialization 취약점 방지**
  - 신뢰되지 않은 직렬화 포맷 객체 역직렬화 금지(특히 native object).

---

## 4. CSRF / CORS / 클릭재킹
- **CSRF 방어**
  - Cookie 기반 세션이면 CSRF 토큰 + SameSite 적절히 설정.
  - 상태 변경(POST/PUT/PATCH/DELETE)은 CSRF 검증 필수.
- **CORS 최소화**
  - `Access-Control-Allow-Origin: *` 지양.
  - Origin allowlist 운영(환경별 분리).
  - 인증정보 포함 요청(`credentials`) 사용 시 더 엄격히.
- **클릭재킹 방어**
  - `X-Frame-Options` 또는 `Content-Security-Policy frame-ancestors` 설정.

---

## 5. 보안 헤더(권장 “필수” 세트)
- **HSTS**: HTTPS 강제 (`Strict-Transport-Security`)
- **CSP**: XSS/리소스 로딩 통제 (`Content-Security-Policy`)
- **X-Content-Type-Options**: `nosniff`
- **Referrer-Policy**: 최소화
- **Permissions-Policy**: 카메라/마이크/위치 등 불필요 기능 차단

---

## 6. 암호화 (전송/저장)
- **전송 구간 암호화**
  - 전 구간 HTTPS/TLS (HTTP는 리다이렉트)
  - 최신 TLS 설정(낡은 프로토콜/취약한 Cipher 제거)
- **저장 시 암호화**
  - 비밀번호는 반드시 **강한 해시(예: bcrypt/argon2 계열)** + salt.
  - API Key/토큰/비밀 값은 KMS/Secret Manager 사용(소스/환경변수 평문 저장 지양).
  - DB/디스크 암호화(가능하면) + 백업도 동일 적용.
- **키 관리**
  - 키/시크릿 로테이션 절차 정의.
  - 접근권한 최소화 및 감사로그.

---

## 7. 민감정보/개인정보 보호
- **데이터 분류**
  - Public / Internal / Confidential / Restricted 등 등급화.
- **노출 최소화**
  - 응답에서 불필요한 필드 제거(특히 PII).
  - 마스킹(예: 이메일 일부, 전화번호 일부).
- **로그에 민감정보 금지**
  - 비밀번호, 세션/리프레시 토큰, API 키, 주민번호/여권 등.
- **보존/파기 정책**
  - 필요 기간만 보관하고 파기(백업 포함).

---

## 8. 파일 업로드/다운로드 보안
- **업로드 검증**
  - 파일 크기 제한, 확장자만 믿지 말고 MIME/시그니처 확인.
  - 실행 가능 파일 차단(스크립트/바이너리 등).
- **저장 위치**
  - 웹 루트 밖 저장(직접 실행/서빙 방지).
  - 파일명은 서버에서 재생성(UUID 등).
- **악성코드 스캔(가능하면)**
  - AV/스캐너 연동 또는 업로드 격리.
- **다운로드**
  - 권한 체크 필수, `Content-Disposition` 설정, 경로 조작 방지.

---

## 9. 에러 처리 & 정보노출 방지
- **에러 메시지 표준화**
  - 사용자에게는 일반 메시지(“요청 처리 실패”) + 내부적으로만 상세 로그.
- **스택트레이스/디버그 페이지 차단**
  - 운영 환경에서 debug off.
- **버전/구성 정보 숨김**
  - 서버 배너, 상세 에러, 프레임워크 정보 최소화.

---

## 10. 로깅, 모니터링, 감사(Audit)
- **보안 이벤트 로깅**
  - 로그인 성공/실패, MFA, 권한 변경, 관리자 작업, 데이터 내보내기, 설정 변경.
- **탐지/알림**
  - 비정상 로그인(국가/시간대/속도), rate limit 초과, 4xx/5xx 급증, WAF 차단 이벤트.
- **감사로그 무결성**
  - 수정 불가 스토리지(append-only) 또는 별도 저장소.
- **개인정보/민감정보 마스킹 후 기록**

---

## 11. Rate Limiting / Abuse 방지
- **API Rate Limit**
  - IP/계정/토큰 단위 제한 + 버스트 제어.
- **Brute-force 방지**
  - 로그인/OTP/비밀번호 재설정/회원가입/검색 API 등.
- **봇/스크래핑 방지(필요 시)**
  - 사용자 행동 기반 제한, CAPTCHA(남용 endpoint 위주).

---

## 12. 의존성(라이브러리) & 공급망 보안
- **의존성 취약점 스캔**
  - SCA 도구 사용(예: npm audit, pip-audit, Snyk 등).
- **버전 고정 & 업데이트 정책**
  - lockfile 유지, 정기 업데이트, 취약점 패치 SLA 정의.
- **빌드 아티팩트 신뢰**
  - CI에서 재현 가능한 빌드, 서명/체크섬(가능하면).
- **불필요 패키지 제거**
  - 사용하지 않는 라이브러리/기능은 제거.

---

## 13. CI/CD & 배포 보안
- **Secret 관리**
  - CI 변수/Secret store 사용, PR 로그/아티팩트에 노출 금지.
- **권한 분리**
  - 배포 권한 최소화, 프로덕션 변경은 승인/리뷰 필요.
- **환경 분리**
  - dev/stage/prod 분리, prod 데이터의 dev 사용 금지(필요 시 마스킹).
- **IaC 보안**
  - Terraform/CloudFormation 등은 코드 리뷰 + 스캔.

---

## 14. 인프라/네트워크 기본 보안
- **WAF/Reverse Proxy**
  - 기본 룰 + 커스텀 룰(로그인/업로드 등).
- **네트워크 분리**
  - DB는 private network, 직접 인터넷 노출 금지.
- **방화벽 정책**
  - 필요한 포트만 허용.
- **백업/복구**
  - 정기 백업, 복구 리허설, 랜섬웨어 대비(오프라인/immutable 백업 고려).

---

## 15. 보안 테스트(릴리즈 게이트)
- **필수 테스트 항목**
  - 인증/인가(권한 상승, IDOR)
  - 인젝션(SQL/NoSQL/Command)
  - XSS/CSRF/CORS
  - 업로드/다운로드
  - Rate limiting/Brute force
  - 민감정보 노출(로그/응답/에러)
- **자동화**
  - SAST(정적 분석), DAST(동적 스캔), SCA(의존성) 최소 1회 이상 파이프라인에 포함.
- **침투 테스트(규모/리스크에 따라)**
  - 외부 노출 서비스/결제/PII 처리 시 정기 수행 권장.

---

## 16. 운영 체크리스트 (Go-Live 직전)
- [ ] 운영환경 debug OFF / 상세 에러 비노출
- [ ] HTTPS 강제 + HSTS 적용
- [ ] 보안 헤더(CSP 포함) 적용
- [ ] CORS allowlist 점검
- [ ] CSRF 방어(세션 기반일 경우) 확인
- [ ] 비밀번호 해시/토큰 정책 확인
- [ ] 관리자 계정 MFA 적용
- [ ] Rate limit 적용(로그인/OTP/검색/업로드 등)
- [ ] 로그 마스킹(토큰/PII) 확인
- [ ] 의존성 취약점 High/Critical 0
- [ ] 백업/복구 절차 문서화 및 테스트
- [ ] 접근권한(Cloud/DB/CI) 최소화 점검

---

## 17. (선택) 프로젝트에 바로 붙이는 “보안 요구사항 문구” 예시
- “모든 API는 서버 측 권한 검증을 반드시 수행한다.”
- “사용자 입력은 allowlist 기반 검증을 적용하며, DB 질의는 parameterized query만 사용한다.”
- “세션/토큰은 HttpOnly/Secure/SameSite 정책을 준수하며, 재인증/만료 정책을 적용한다.”
- “로그에는 비밀번호/토큰/키/민감정보를 기록하지 않는다.”
- “배포 파이프라인에 SAST/SCA를 포함하고, High/Critical 취약점이 존재하면 릴리즈를 차단한다.”
